Référence API Explorer

Tous les endpoints API côté serveur disponibles dans l’Explorer jinflow.

Ce sont des API internes de l’Explorer, pas une API REST publique. Elles servent l’interface web de l’Explorer. Documentées ici pour la transparence et pour les équipes construisant des intégrations.

Pour la référence complète des endpoints avec tous les paramètres et formats de réponse, voir la version anglaise.

Catégories d’endpoints

Section intitulée « Catégories d’endpoints »

Santé et statut

Section intitulée « Santé et statut »
  • GET /health — vérification de santé avec statut de connectivité DB
  • GET /api/identity — identité git de l’utilisateur courant
  • GET /api/store-status — progression de téléchargement du KLS store (mode R2)
  • GET /api/db-status/[tenant] — sondage léger mtime/generation

Recherche

Section intitulée « Recherche »
  • GET /api/jumpbar — recherche globale sur tous les types de contenu (modes : search, tenants, summary)

Livraison de contenu

Section intitulée « Livraison de contenu »
  • GET /api/logo/[tenant] — image logo du tenant
  • GET /api/pipeline/[tenant]/[artifact_key] — artefact DAG pipeline depuis le KLS
  • GET /api/reports/[tenant]/[report_key] — blob PDF de rapport

Lineage et explication

Section intitulée « Lineage et explication »
  • GET /api/explain — Niveau 2 : enregistrements Gold source qui ont contribué à un Finding
  • GET /[tenant]/api/explain/[finding_id] — Niveau 1+2 : métadonnées Finding + enregistrements source
  • POST /api/explain-narrative — Niveau 4 : explication en langage naturel générée par IA

Détection de drift

Section intitulée « Détection de drift »
  • GET /api/drift — comparer le commit de build KLS avec le HEAD du remote GitHub

KLS Store

Section intitulée « KLS Store »
  • POST /api/refresh — invalider les fichiers R2 cachés et déclencher un re-téléchargement

Vues sauvegardées

Section intitulée « Vues sauvegardées »
  • GET /api/views — lister les vues sauvegardées
  • POST /api/views — créer une nouvelle vue
  • PUT /api/views — mettre à jour une vue existante
  • DELETE /api/views — supprimer une vue

Signal Builder

Section intitulée « Signal Builder »
  • GET /api/probe-builder/list — lister les définitions de Signal disponibles
  • GET /api/probe-builder/load — charger le YAML source d’un Signal
  • POST /api/probe-builder/save — écrire le YAML Signal dans l’AFS
  • POST /api/probe-builder/validate — valider le YAML Signal contre les Contracts
  • POST /api/probe-builder/compile — compiler le YAML Signal en SQL

Notebooks

Section intitulée « Notebooks »
  • GET /[tenant]/api/notebook — obtenir le notebook du tenant
  • POST /[tenant]/api/notebook/cells — ajouter une cellule
  • PUT /[tenant]/api/notebook/cells/[cell_id] — mettre à jour une cellule
  • DELETE /[tenant]/api/notebook/cells/[cell_id] — supprimer une cellule
  • POST /[tenant]/api/notebook/cells/[cell_id]/discussion — ajouter une réponse à un fil de discussion

Notes

Section intitulée « Notes »
  • GET /[tenant]/api/notes — lister toutes les notes
  • POST /[tenant]/api/notes — créer une note
  • GET /[tenant]/api/notes/[note_id] — obtenir une note
  • POST /[tenant]/api/notes/[note_id] — ajouter cellule ou réponse
  • PUT /[tenant]/api/notes/[note_id] — mettre à jour une note ou cellule
  • DELETE /[tenant]/api/notes/[note_id] — supprimer une note ou cellule

Taxonomie et dimensions

Section intitulée « Taxonomie et dimensions »
  • GET /[tenant]/taxonomies/[taxonomy_id]/members — lister les membres de taxonomie
  • GET /[tenant]/taxonomies/cardinality/member — mappings inter-taxonomies pour un membre
  • GET /[tenant]/taxonomies/cardinality/members — membres partagés entre deux noeuds
  • GET /[tenant]/taxonomies/cardinality/fanout — distribution de fanout
  • GET /[tenant]/taxonomies/cardinality/detail — métriques de cardinalité
  • GET /[tenant]/dimensions/[entity]/export — export CSV d’une table de dimension
