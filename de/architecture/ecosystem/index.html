Das jinflow-Oekosystem

Uebersicht

Abschnitt betitelt „Uebersicht“

jinflow ist eine gouvernierte Analytics-Engineering-Plattform. Sie trennt Engine von Domain-Wissen von Daten von kompiliertem Output, jeweils in einem eigenen Repository mit eigenem Lebenszyklus.

Die fuenf Stores

Abschnitt betitelt „Die fuenf Stores“
StoreVollstaendiger NameWas er haeltLebenszyklus
DPRDomain Pack RepositoryDomain-Blueprint: Contracts, dbt-Modelle/Macros, Dispatch, Basisinstrumente, GlossarVon Domain-Experten gepflegt. Versioniert.
AFSArtifact StoreTenant-spezifische Instrumente: Signals, Theses, Verdicts, SMEbits, BitBundlesAus DPR gebootstrapped, waechst dann unabhaengig.
DLZData Landing ZoneRohe Kundenlieferungen: Excel-Exporte, CSVsVersioniert in Git. Nur-Anfuegen.
KLSKnowledge StoreKompilierte DuckDB-Datenbank: Bronze bis Gold, Signal Findings, Thesis Verdicts, Verdict FindingsUnveraenderlich nach Build.
SISSystem Information StoreBuild-Journal, Instrumentenlebenszyklus, KompilierungsprotokollVeraenderlich, akkumuliert ueber Builds.

Der Build-Treffpunkt

Abschnitt betitelt „Der Build-Treffpunkt“

jinflow make ist der einzige Punkt wo Wissen auf Daten trifft:

  • AFS liefert was gesucht werden soll — Instrumente, Contracts, analytische Logik
  • DLZ liefert worin gesucht werden soll — Rohdaten aus Quellsystemen des Kunden
  • KLS ist was herauskommt — ein eigenstaendiges, abfragbares analytisches Produkt
  • SIS zeichnet auf was passiert ist — Build-Provenienz, Zeitstempel, Instrumentenzustaende
