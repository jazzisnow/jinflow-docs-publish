Probe System — Full Lifecycle

Dieser Inhalt ist noch nicht in deiner Sprache verfügbar.

Date: 2026-02-20

This document describes the jinflow probe system end-to-end: from detection through assessment, interpretation, prioritisation, action, resolution, and verification.

The guiding metaphor (from the DDD glossary): the organization is the patient, probes are diagnostics, findings are symptoms, assessments are differential diagnoses, interpretations explain in plain language, treatments are interventions, and re-probing closes the loop.

Overview

Section titled “Overview”

Each phase has a clear input, output, and owner. Phases 1-3 are automated. Phases 4-7 are progressively more human-in-the-loop today, but designed for automation as the system matures.

Phase 1 — Detect

Section titled “Phase 1 — Detect”

Goal: Produce standardised findings from data layer tables.

Input: Gold layer tables (the product contract) for financial_anomaly and compliance probes. Silver layer tables for data_quality probes that need to see invalid rows.

Output: probe_findings__<probe_id> tables, each row a finding.

Input layer by category

Section titled “Input layer by category”
CategoryInput layerWhy
financial_anomalyGoldOperates on validated, deduplicated business entities
data_qualitySilver or GoldSilver when checking validation rates / invalid rows (Gold filters them out). Gold when checking master data completeness (e.g. supplier coverage)
complianceGoldRegulatory checks on validated data

Section titled “Finding output contract”

Every probe emits a table with these columns:

Probe types

Section titled “Probe types”
  • DSL probes (declaratively defined): balance, mandatory_item, distribution_outlier. Compiled to SQL by probecompile.py. For repeatable patterns.
  • Hand-written probes (SQL): For complex joins or custom logic that don’t fit the DSL. Same output contract, same models/probes/ location.

Section titled “Dynamic discovery”

The platform and Explorer discover probes at runtime by querying information_schema.tables for tables matching probe_findings__% in each tenant schema. No hardcoded probe lists.

Silver is_valid semantics

Section titled “Silver is_valid semantics”

Silver models use is_valid (boolean) and invalid_reason (varchar) to flag rows that fail validation. Gold filters to is_valid = true. Understanding what is_valid means is essential for data_quality probes.

Three categories of validation:

CategoryExamplesSemantics
Missing PK / required fieldmissing case_token, missing material_id, missing supplier_codeRow is structurally unusable — cannot participate in any join or aggregation. Always fatal.
Invalid valueinvalid quantity (null/≤0), invalid case_type (not in enum), invalid movement_type, invalid billing_statusDomain logic cannot process this row. The value exists but is outside the allowed domain. Fatal.
Orphan FKorphan case_token, orphan material_id, orphan parent_node_idThe referenced parent entity is missing. Ambiguous — could mean: (a) source data is incomplete, (b) parent was purged, (c) parent failed its own validation (cascading invalidity).

The first two categories are clear-cut: the row is broken. The third is where data_quality probes add value — not by re-checking individual rows, but by analyzing the aggregate pattern: What percentage of billing events have orphan case_tokens? Is it increasing? What’s the monetary impact of excluded rows?

is_valid is a validation gate (per-row, binary, deterministic). Data quality probes are diagnostics (aggregate, severity-graded, trend-aware). They complement each other — probes don’t replace is_valid, they analyze its output.

Phase 2 — Assess

Section titled “Phase 2 — Assess”

Goal: Aggregate symptoms into entity health scores. Probes detect individual symptoms; assessments tell you how healthy an entity is overall.

Input: Findings from multiple probes for a tenant.

Output: Per-entity assessment findings with severity, health_score, and aggregated risk.

The problem assessments solve

Section titled “The problem assessments solve”

Individual findings tell you what is wrong. But the same entity can be flagged by many probes. Without assessments, the user sees a flat list of symptoms and must mentally aggregate the overall picture per entity.

Assessment architecture

Section titled “Assessment architecture”

Assessments are a second-order construct: they consume findings, not raw data. They run after all probes complete and emit the same findings contract.

Assessment output

Section titled “Assessment output”

Assessments emit the standard findings contract with these additional computed fields available in the aggregated CTE:

FieldTypeDescription
probe_countintNumber of distinct probes flagging the entity
finding_countintTotal findings across all source probes
total_riskdecimalSum of money_at_risk from all findings
weighted_riskdecimalRisk weighted by source probe weights
weighted_probe_scoredecimalSum of source probe weights
health_scoredecimal0.0 (all probes flag) to 1.0 (no flags)
worst_severityvarcharHighest severity across source probes
probes_flaggedvarcharComma-separated probe_ids

Phase 3 — Interpret

Section titled “Phase 3 — Interpret”

Goal: Translate a finding into a human-readable explanation in the user’s language, so non-technical stakeholders understand what happened and why it matters.

Input: A finding row (including evidence JSON).

Output: A localised text (DE/FR/EN) explaining the finding.

Interpretation templates

Section titled “Interpretation templates”

Each probe defines its own interpretation templates, embedded in the probe definition. The interpreter performs string substitution — no code changes needed for new probes.

Template structure per probe:

Placeholders are resolved from the finding’s evidence JSON plus the standard finding columns (entity_id, severity, money_at_risk, time_bucket).

Fallback

Section titled “Fallback”

If a probe has no registered template, the interpreter renders the evidence JSON as a structured key-value list. Functional but not polished — an incentive to always provide templates.

Phase 4 — Prioritise

Section titled “Phase 4 — Prioritise”

Goal: Rank findings so users focus on what matters most.

Input: All findings for a tenant (or across tenants).

Output: A ranked, filterable list with context.

Prioritisation signals

Section titled “Prioritisation signals”
SignalSourceWeight
severityProbe-assigned (high/medium/low)Primary
money_at_riskProbe-computedPrimary
trend_directionTrend probes (worsening > stable > improving)Secondary
RecurrenceSame entity flagged across multiple probesSecondary
AgeHow long the finding has persistedTertiary

Prioritisation today

Section titled “Prioritisation today”

The Explorer sorts by money_at_risk DESC within severity groups. This is adequate for v1. Future: a composite priority score combining the signals above.

Cross-probe correlation

Section titled “Cross-probe correlation”

A material flagged by both io_coefficient (bad level) and io_coefficient_trend (getting worse) is a stronger signal than either alone. The prioritisation layer should surface these clusters.

Phase 5 — Act

Section titled “Phase 5 — Act”

Goal: Recommend or trigger concrete next steps for a finding.

Input: A prioritised finding with interpretation.

Output: A recommended action or set of actions.

Action types

Section titled “Action types”
ActionDescriptionAutomation level
InvestigateGather more data — drill into the entity, check related findingsManual today, guided navigation in Explorer
EscalateRoute to a responsible person or teamManual today
CorrectFix the root cause (e.g. adjust stock levels, rebill, update CC mapping)Always manual — the Explorer is read-only
AcceptAcknowledge and accept the risk (with justification)Manual with audit trail
SuppressMark as known/expected (e.g. seasonal pattern) with expiryManual today, rule-based later

Phase 6 — Resolve

Section titled “Phase 6 — Resolve”

Goal: Track the disposition of each finding through its lifecycle.

Input: A finding + human decision.

Output: A resolution record.

Resolution states

Section titled “Resolution states”

Resolution record

Section titled “Resolution record”
FieldDescription

Where resolution state lives

Section titled “Where resolution state lives”

Findings are recomputed on every dbt build (immutable, deterministic). Resolution state is mutable and lives outside dbt — in a separate resolutions table managed by the Explorer app or a future API layer. Joined at query time.

A finding that disappears after a dbt rebuild (because the underlying data improved) is auto-resolved.

Phase 7 — Verify

Section titled “Phase 7 — Verify”

Goal: Confirm that actions taken actually fixed the problem.

Input: A resolved finding + the next probe run.

Output: Verification status (confirmed fixed, regressed, unchanged).

Verification logic

Section titled “Verification logic”

After a finding is marked resolved:

  1. The next dbt build re-runs all probes
  2. If the finding_id no longer appears → confirmed fixed
  3. If it reappears with lower severity or lower money_at_risk → improving
  4. If it reappears unchanged or worse → regressed (auto-reopen)

Temporal tracking

Section titled “Temporal tracking”

Because finding_id is deterministic (hash of probe + tenant + entity + time_bucket), the same finding across builds can be tracked over time. A finding_history table records:

FieldDescription
finding_idThe finding
build_timestampWhen the dbt build ran
severitySeverity at this point in time
money_at_riskImpact at this point in time

This enables trend-over-time views: “this finding has been open for 3 builds, money_at_risk is increasing.”

How to register a new probe

Section titled “How to register a new probe”

Follow this checklist when adding a new probe:

1. Write the probe SQL

Section titled “1. Write the probe SQL”

Create models/probes/probe_findings__<probe_id>.sql:

  • Materialized as table, tagged probe: {{ config(materialized='table', tags=['probe']) }}
  • Reference Gold entities: {{ ref('gold_...') }}. Data quality probes may also reference Silver: {{ ref('silver_...') }}
  • Emit the standard findings columns: finding_id, tenant_id, probe_id, probe_version, severity, entity_type, entity_id, time_bucket, money_at_risk, evidence
  • finding_id must be deterministic: md5('probe_id' || '|' || tenant_id || '|' || entity_id || '|' || time_bucket)
  • evidence is a JSON string with probe-specific details

2. Register in probe_registry

Section titled “2. Register in probe_registry”

The registry is auto-generated by proberegistry.py. Two options:

Option A (auto-generated): Just run python3 scripts/proberegistry.py. The tool reads your probe YAML and generates display_name, description, and interpretation templates automatically from the DSL structure + glossary.

Option B (override): Add a registry: block to your probe YAML with custom tri-lingual text, then run proberegistry.py:

Placeholders ({...}) are resolved from evidence JSON keys and standard finding fields (entity_id, entity_type, money_at_risk, severity, time_bucket, probe_id). See docs/architecture/probe_registry.md for the full override mechanism.

3. Add to platform_probe_findings

Section titled “3. Add to platform_probe_findings”

Edit models/platform/platform_probe_findings.sql:

  • Add a -- depends_on: {{ ref('probe_findings__<probe_id>') }} comment
  • Append 'probe_findings__<probe_id>' to the probe_models list

(This is a dbt limitation — ref() must be statically parseable.)

4. Document in YAML

Section titled “4. Document in YAML”

Add an entry to models/probes/probes.yml with description and column docs.

5. Build and verify

Section titled “5. Build and verify”

6. What you do NOT need to change

Section titled “6. What you do NOT need to change”
  • Explorer TypeScript — probe discovery is dynamic (information_schema)
  • Interpretation rendering — templates come from probe_registry

Design principles

Section titled “Design principles”
  1. Probes are self-describing — carry their own metadata, interpretation templates, and recommended actions. No central registry that must be updated.
  2. Detection is immutable — findings are recomputed fresh each build. Mutable state (resolution, assignment) lives in a separate layer.
  3. Input layer matches the question — financial_anomaly and compliance probes read Gold (validated business entities). Data quality probes may read Silver (to see what Gold filtered out). Bronze is never accessed.
  4. Assessments consume findings, not data — assessments are second-order: they correlate symptoms into diagnoses. They never query Gold or Silver directly.
  5. Dynamic discovery — new probes and assessments are picked up automatically. No hardcoded lists.
  6. Human-in-the-loop by default — jinflow diagnoses, humans decide and act. Automation is additive, not mandatory.
  7. Feedback closes the loop — the system learns whether interventions worked by re-probing and comparing.
